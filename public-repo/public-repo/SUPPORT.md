# Support

Spec-To-Ship is maintained as a real open-source project.

The software itself is free and intended to remain genuinely useful without paid access. Financial support helps fund maintenance, documentation, issue triage, release work, and continued improvement.

## What this project offers publicly

- Public source code and documentation
- Public issue tracking for bugs and feature requests
- Community-friendly contribution path
- Roadmap visibility through `ROADMAP.md` and release notes through `CHANGELOG.md`

## What support channels are available

### 1. Financial support

If Spec-To-Ship is useful to you or your team, the best way to support it is through the funding options listed in `.github/FUNDING.yml` and the repository sponsor button.

Support helps sustain:

- maintenance and bug fixes
- documentation and examples
- release quality and polish
- roadmap velocity

### 2. Community support

You can also help by:

- opening high-quality bug reports
- proposing focused feature requests
- improving docs and examples
- contributing fixes through pull requests
- sharing the project with people who may benefit from it

### 3. Professional help around the project

For teams that want direct help adopting or extending Spec-To-Ship, project-related services may be offered separately, such as:

- setup and enablement sessions
- workflow design guidance
- architecture reviews
- internal pilot workshops
- integration scoping

These services are optional and separate from sponsorship. Sponsorship supports the project; it does not purchase private ownership of the roadmap.

## Support boundaries

To keep the project healthy and credible:

- sponsorship does **not** guarantee feature delivery on demand
- sponsorship does **not** grant private support SLAs by default
- sponsorship does **not** give sponsors unilateral roadmap control
- critical bug fixes and project health decisions are made in the best interest of the project as a whole

If private advisory, workshops, or implementation help are offered, they will be scoped explicitly as separate professional services.

## Where to ask for help

- **Bug reports / feature requests:** GitHub Issues
- **Security concerns:** see `SECURITY.md`
- **Contributions and development workflow:** see `CONTRIBUTING.md`
- **Project direction and decision model:** see `GOVERNANCE.md`

## Response expectations

Spec-To-Ship is maintained on a best-effort basis.

That means:

- issues and pull requests are reviewed as time allows
- urgent security matters are handled through the security process
- roadmap work is prioritized based on project impact, coherence, maintenance cost, and available capacity

If you rely on the project professionally, financial support or a separately scoped services engagement can help increase the time available for maintenance and guidance.
