# Governance

This document describes how decisions are made for Spec-To-Ship.

The goal is to keep the project:

- technically coherent
- useful as a real open-source tool
- maintainable over time
- fair to contributors and users

## Project philosophy

Spec-To-Ship is an open-source engineering tool that should:

- remain free and genuinely useful
- prioritize clarity, structure, and execution quality
- avoid unnecessary complexity or over-engineering
- serve both individual developers and professional teams

## Maintainer model

The project is currently maintained by the core maintainer.

The maintainer is responsible for:

- architectural direction
- reviewing and merging pull requests
- release decisions
- roadmap prioritization
- maintaining project coherence

## Contributions

Contributions are welcome and encouraged.

All contributions must:

- follow the guidelines in `CONTRIBUTING.md`
- pass all required checks
- align with the project’s direction and quality standards

Non-trivial changes should be discussed in an issue before implementation.

## Decision-making principles

When evaluating changes or proposals, decisions are guided by:

- long-term maintainability
- simplicity and clarity
- alignment with the project’s core workflow model
- real-world usefulness
- cost of complexity vs. benefit

Not all valid ideas will be accepted. Rejections are not a judgment of quality, but of fit.

## Roadmap ownership

The roadmap is maintained in `ROADMAP.md`.

The maintainer defines priorities based on:

- user impact
- project coherence
- maintenance cost
- available time and resources

Community feedback is considered but does not override core direction.

## Sponsorship and governance

Financial support does **not** grant:

- control over the roadmap
- guaranteed feature delivery
- privileged decision-making power

Sponsors may:

- receive recognition
- receive periodic updates
- participate in higher-level discussions (depending on tier)

But all final decisions remain aligned with the project’s long-term health.

## Services vs. open source

Professional services (if offered) are separate from the open-source project.

This means:

- the OSS project remains fully usable without payment
- services are optional and clearly scoped
- no critical functionality is hidden behind paid access

## Conflict resolution

In case of disagreement:

- discussions should remain constructive and focused on the project
- maintainers may make final decisions when needed
- contributors are free to fork the project if a different direction is desired

## Evolution

This governance model may evolve as the project grows.

Any major changes will be reflected in this document and communicated through the repository.
